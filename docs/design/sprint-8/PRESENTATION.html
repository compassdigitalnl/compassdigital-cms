🎯 Implementatie Opties

We bieden twee implementatie-varianten, afhankelijk van hoe subscribers zich kunnen authenticeren.

Optie A: Basis Implementatie

Voor wie? Subscribers die zich registreren op de Aboland webshop krijgen een account.

User Flow:

  1. Subscriber bezoekt aboland.nl/my-account
  2. Kiest "Registreren" en maakt handmatig een account aan
  3. Account wordt gekoppeld aan THOR abonnementen (via email match)
  4. Kan nu inloggen en abonnementen beheren

Kenmerken:

  • Standaard email/wachtwoord authenticatie
  • Persistent account met wachtwoord
  • Database blijft klein (alleen geregistreerde users)

Optie B: Magic Link + Auto-Account ⭐

Voor wie? Alle subscribers kunnen inloggen via een magic link, zonder handmatige registratie. Bij eerste gebruik wordt automatisch een persistent account aangemaakt.

User Flow (Eerste Keer):

1
Subscriber vult gegevens in
Email + postcode + huisnummer op aboland.nl/abonnement-inzien
2
THOR API verificatie
Systeem checkt THOR: abonnement gevonden? Adres klopt?
3
Magic link verzonden
Email met inloglink (24 uur geldig) wordt verstuurd
4
Account wordt automatisch aangemaakt ✨
Bij klik op link: account creatie + alle abonnementen gekoppeld
5
Subscriber is ingelogd
Dashboard met alle abonnementen, facturen en beheer mogelijkheden

Vanaf Nu (Volgende Bezoeken):

  • Optie 1: Gewoon inloggen met email/wachtwoord (persistent account aanwezig!)
  • Optie 2: Opnieuw magic link gebruiken (altijd mogelijk)
✨ Beste van Beide Werelden:
Frictionless start (geen registratie formulier) + persistent account voor toekomstig gebruik. Subscriber hoeft maar 1x magic link te gebruiken, daarna kan normal inloggen.

🏗️ Technische Architectuur

3-Tier Structuur

Tier 1: THOR API (Source of Truth)
Alle abonnementsdata leeft in THOR. THOR is en blijft de enige bron van waarheid voor subscriptions, invoices en magazines. Payload haalt data op-demand op via THOR REST API V2.
Tier 2: Payload CMS (Middleware & Cache)
Payload fungeert als middleware en cache layer. Cached subscription data voor performance, maar THOR blijft source of truth. Nieuwe database tables: thor_subscriptions, thor_invoices, thor_magazines, thor_actions, thor_sync_log.
Tier 3: Next.js Frontend (Webshop)
React componenten voor subscriber self-service. Volledig headless: frontend kan vervangen worden zonder architectuur te wijzigen. Routes: /my-account/subscriptions, /my-account/subscription/[id], /my-account/invoices.

Data Flow

→
User bezoekt /my-account/subscriptions
Next.js frontend laadt subscriber dashboard
→
API call naar Payload
Frontend vraagt subscription data op via REST API
→
Payload checkt cache (15 min TTL)
Als data < 15 min oud: return cached data
→
THOR API sync (indien nodig)
Als cache expired: fetch fresh data from THOR + update cache
→
Return data naar frontend
Subscriptions + invoices worden gerenderd in UI

🗄️ Database Schema

Nieuwe Payload collections voor THOR data caching:

📦
thor_subscriptions
Cached subscription data: magazine, looptijd, status, bezorgadres, betaalgegevens, ontvanger vs betaler. Foreign key naar users.id.
🧾
thor_invoices
Facturen per subscription: bedrag, status (betaald/open), vervaldatum, betaalreferentie, download URL.
📰
thor_magazines
Magazine metadata: titel, uitgever, publicatietype (print/digital/combi), omschrijving.
🎯
thor_actions
Beschikbare abonnementsacties/aanbiedingen vanuit THOR voor upselling.
🔄
thor_sync_log
Audit trail: wanneer werd data gesynchroniseerd, welke errors, hoeveel records, sync duration.
👤
users (extended)
Extra velden voor Optie B: emailVerified, accountSource, thorVerifiedAt, lastLoginAt, createdVia.
⚠️ Belangrijk: THOR blijft de source of truth. Payload tables zijn read-only cache. Wijzigingen worden via THOR API doorgestuurd, niet direct in database.

🔄 Synchronisatie Strategie

2-Way Communication

↓
THOR → Payload (Read)
Ophalen van subscriptions, invoices, magazines via GET endpoints. Cache TTL: 15 minuten. Background sync: elke 30 minuten voor actieve users.
↑
Payload → THOR (Write)
Adreswijzigingen, opzeggingen, email updates via POST/PUT/PATCH endpoints. Direct naar THOR gestuurd, daarna cache ge-update.

Lazy Loading Principe

Efficient & Schaalbaar: We synchroniseren NIET alle 6+ miljoen subscribers uit THOR. Alleen subscribers die de webshop daadwerkelijk gebruiken worden toegevoegd aan de database (50K-100K). Dit houdt de database klein, queries snel en kosten laag.

Wanneer wordt een user toegevoegd?

  • Optie A: Wanneer user zich registreert
  • Optie B: Wanneer user magic link gebruikt (eerste keer)
  • Background sync: Alleen users die <30 dagen geleden ingelogd zijn

Automatische Cleanup

Users die 12+ maanden inactief zijn EN geen actief abonnement meer hebben, kunnen automatisch worden verwijderd. Als ze later terugkomen wordt het account opnieuw aangemaakt met verse THOR data.

🔒 Security & GDPR

🔐
JWT Authentication
THOR API authenticatie via JWT bearer tokens. Tokens worden gecached met expiry check. Automatische refresh bij verlopen tokens.
🛡️
Rate Limiting
THOR API rate limit: 100 requests per 5 seconden per endpoint. Client-side rate limiter voorkomt overschrijding.
🔍
Input Validation
Alle user input wordt gevalideerd (email, postcode, huisnummer). XSS en SQL injection preventie via Payload's ingebouwde sanitization.
📜
GDPR Compliant
Recht op inzage (dashboard), recht op verwijdering (account deletion), recht op correctie (adres/email wijzigen). Audit trail via sync logs.
🔒
API Key Security
THOR credentials opgeslagen in environment variables (.env), nooit in code. Server-side only, nooit blootgesteld aan frontend.
✅
Authorization Checks
Users kunnen alleen hun eigen subscriptions/invoices zien. Payload access control policies op collection-niveau.
